# 架構與工程優化建議

> 版本：1 ｜ 更新日期：2026-07-15 ｜ 狀態：current
> 對應缺口：[../02-gaps.md §4、§6、§9、§15、§17](../02-gaps.md)
> 目的：整理目前設計文件的跨系統風險，並給出進入 PoC 前後的優化優先序

---

目前專案仍處於設計文件階段，尚無應用程式碼與自動化測試。現有設計已涵蓋法案、時間、媒體、事件注入、人物 profile、成本與可觀測性，但還不適合直接進入完整產品開發。建議先完成一個「可重現、可驗證、單點失敗不會卡死」的垂直 PoC。

## P0：實作前優先處理

### 1. 定義最小容錯與動作驗證合約（缺口 #9）

完整的成熟版容錯可以延後，但 PoC 至少要具備：

- LLM 輸出以 discriminated union schema 驗證。
- 拒絕不存在的 ID 與非法 stage transition。
- 為 timeout、格式錯誤定義有限次 retry 與 deterministic fallback。
- 每個動作帶 `idempotencyKey`，避免重送造成重複寫入。
- 非同步結果帶 generation/version，回傳時若 world generation 已改變就丟棄 stale result。
- 單一 Agent 失敗只標記該動作失敗，不得中斷整個 tick。

### 2. 補齊可重現性合約（缺口 #17）

[time-model.md](../design/time-model.md) 已假設「給定 seed 和輸入序列即可重現」，但尚未定義 seed 如何傳播。建議釘下：

- 根種子 `simulationSeed`。
- Agent、媒體、民調等子系統的衍生 seed。
- 固定 Agent 與事件的執行順序。
- 外部輸入採 append-only event log。
- 保存 prompt 版本、模型 ID、模型參數、profile hash。
- fork 明確記錄來源 snapshot 與 event offset。
- LLM 無法保證 bit-for-bit deterministic 時，保存原始回應供 replay。

`temperature=0` 只能降低隨機性，不應作為完全可重現的保證。

### 3. 統一跨模型 Canonical Topic Registry

[bill-model.md](../design/bill-model.md) 的 `stanceVector` 目前只有 `economic / environment / social`，但 [time-model.md](../design/time-model.md) 的 λ 分類還包含 `core_ideology / constitutional / identity / tactical` 等維度。

建議建立唯一 `TopicRegistry`，集中管理：

- topic ID、分類與 stance 維度。
- stance decay λ。
- 法案投影權重。
- 媒體、民調與 profile 的 topic 映射。
- rule-pack 的擴充與覆寫方式。

這可避免同一議題在不同子系統使用不同名稱，導致資料無法 join 或公式錯套。

### 4. 建立最小 ground truth 評估集（缺口 #6）

目前 `behavior_consistency_score` 用系統計算的 `naturalLean` 評估系統自己的投票，容易形成循環驗證：規則式捷徑越多，分數可能越漂亮，但不代表更貼近真實政治行為。

建議拆成三類指標：

- 歷史投票預測：accuracy、F1、Brier score。
- 行為穩定性：相同輸入多次執行的結果分布差異。
- 反事實敏感度：只修改單一壓力變數時，結果是否呈合理、可解釋的變化。

資料需按時間切分 calibration/test，避免使用未來資料建立 profile 後再回測過去。

### 5. 選民層先用統計 cohort（缺口 #4）

PoC 不需要建立大量 LLM 選民 Agent。建議用 per-`(district, demographic, topic)` 的統計 cohort，至少包含：

- baseline stance distribution。
- turnout propensity。
- media receptivity。
- party affinity。
- economic sensitivity。

這也能補上 [media-model.md](../design/media-model.md) 目前未定義的 `baseline` 與 `district_receptivity`。

## P1：PoC 架構優化

### 6. 採用事件溯源作為核心

建議資料流統一為：

```text
Command → Validate → Domain Event → Reducer → World Snapshot
```

這能同時支援稽核、fork、重播、admin override、除錯與 stale-write 防護。[external-event-model.md](../design/external-event-model.md) 的 admin override 也不應直接改表，而應寫入不可變的 override event，再由 reducer 套用。

### 7. 法案與修正案使用 immutable revision

不建議修正案通過後直接修改 `articles`。每次修正應產生新 revision，保存：

- `rootBillId`。
- `revision`。
- `parentRevisionId`。
- amendment set。
- canonical content hash。
- effective stage。

法案 `stage` 應由有限狀態機管理，所有轉移交給 institutional rule-pack 驗證。

### 8. 修正數值模型的邊界問題

- stance 衰減目前朝 `0` 回歸，應確認是否要改為朝人物 hidden/baseline stance 回歸。
- `evidence_pressure` 加總後應 normalize 並 clip，避免 stance 越界。
- `media_spotlight` 是多則新聞加總，需使用飽和函數避免無上限成長。
- 廣告效果不應直接乘原始貨幣值，應先轉為標準化曝光量。
- `urgency = 1 / max(ticks_to_election, 30)` 在選舉前 30 ticks 完全不變，需確認是否符合原意。

每個公式應加入 invariant、邊界案例與 property-based tests。

### 9. 成本模型改為可更新的 Provider Registry

[llm-cost-model.md](../design/llm-cost-model.md) 的模型 ID、定價和 cache TTL 都是時效性資料，不宜寫死成長期架構假設。建議：

- 建立 provider capability registry。
- 定價資料帶 `effectiveFrom`。
- 每次呼叫保存實際 usage、provider request ID 與 pricing version。
- 預算估算輸出 low/base/high 三種情境。
- 用實測 cache hit rate、活躍 Agent 比例持續回填估算。

### 10. 倫理與安全不能只依靠 disclaimer（缺口 #15）

真實政治人物模式還需具備：

- profile 來源、版本與證據鏈。
- 清楚區分史實、模型推論與虛構事件。
- 高風險或灰色動作的輸出政策。
- 匯出、分享及原始 LLM 輸出的權限控制。
- 外部事件與 profile 內容的 prompt-injection 防護。
- 個人資料、名譽風險與資料保留政策。
- 研究模式與娛樂模式隔離。

## P2：文件與開發流程

### 11. 修正文件一致性

- [README.md](../README.md) 寫「18 項待補」，實際編號已到 19。
- `drafBill()` 應統一為 `draftBill()`。
- 可觀測性文件的「三張表」與實際 schema 數量應統一。
- 行號型交叉引用容易漂移，應逐步改為 heading anchor。
- 各設計檔的「待決事項」應集中成可排序 backlog，標註優先級、owner 與決策期限。
- `.understand-anything/` 若只是本機分析產物，應加入 `.gitignore`，避免誤提交。

### 12. 建立最小工程骨架

第一個 milestone 建議只包含：

- 5 個 Agent。
- 1 個 institutional rule-pack。
- 1 個法案與 immutable revisions。
- 每個 session 最多 20 ticks。
- deterministic event replay。
- 規則式媒體。
- mock LLM adapter 與真實 LLM adapter。
- schema、reducer、replay、錯誤隔離與成本上限測試。
- 一個 CLI runner；前端延後。

PoC 的首要工程驗收標準應是：「相同 event log 可以重建相同 WorldState」。

## 建議執行順序

1. 最小容錯合約。
2. 可重現的事件核心。
3. Canonical Topic Registry。
4. Ground truth 評估集。
5. 5-Agent 垂直 PoC。

這個順序優先處理會造成系統返工的基礎風險，同時避免在核心假設尚未驗證前擴大完整產品設計。
