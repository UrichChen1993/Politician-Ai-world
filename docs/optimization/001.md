1. 最大的風險：規劃順序倒了
01-research.md:355 自己寫「選項 C（PoC）推薦起點」，PoC 場景只是「5 個 Agent + 1 部法案 + 3 輪辯論投票」（01-research.md:508-522）。但目前已經寫了 5 份完整子系統設計文件，卻沒有任何跡象顯示 PoC 真的跑過一次。

問題：繼續補未解缺口（還剩 9 項）是在「還沒驗證核心假設」的情況下疊加設計成本。核心假設（知識圖譜記憶能否影響投票行為）沒驗證，後面設計的都可能要重來。

建議：先用現有已解決的 3 項（bill/time/media model）+ 5 個 Agent 手刻 PoC，其餘缺口先不補，等 PoC 跑完看哪些缺口是真的擋路、哪些是自己想像出來的。

2. bill-model.md 相對 PoC 範圍過度設計
bill-model.md:41-60 有 committee review rounds、版本鏈（parentBillId + version）、cosponsors 等完整國會流程欄位。但 PoC 只需要「一部法案、直接投票」，不需要委員會路徑或修正案家族。

建議：這份設計留著沒關係（供未來參考），但 PoC 實作時只挑 stanceVector + stage（drafted/passed/rejected 三態夠用）+ articles，其餘欄位到真正要做委員會流程時再啟用。不要為了「文件已經設計好」而在 PoC 就把全部欄位建出來。

#13 資訊可見性模型

#15 倫理／責任邊界
02-gaps.md:145-151 指出這模擬真實在世政治人物，發布前一定會被問。這項跟 PoC 進度無關（不影響工程驗證），但成本極低（一份 disclaimer 文件）、風險是名譽/公關層級，不應該用「還在 PoC 階段」當延後理由。
